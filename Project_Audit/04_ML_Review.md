# 04. Machine Learning Review

This review critiques the machine learning methodology employed in the repository.

## 1. Data Leakage (The Cardinal Sin)
The most severe flaw in the pipeline is located in `3_cross_asset_validation.py` (Lines 115-120):
```python
scaler = StandardScaler()
X_scaled = scaler.fit_transform(X)
split_idx = int(len(X) * 0.8)
X_train, X_test = X_scaled[:split_idx], X_scaled[split_idx:]
```
Applying standard scaling over the entire dataset prior to splitting leaks the global mean and variance into the training set. The model is implicitly exposed to future market regimes. Scaling must strictly be `fit` on the training set and `transform` applied to the test set.

## 2. Target Leakage & Intra-Candle Look-Ahead Bias
The target definition mechanism in `3_cross_asset_validation.py` checking future bars (Lines 94-100) exhibits a severe optimistic bias:
```python
if highs[idx + step] >= tp:
    hit_target = True
    break
if lows[idx + step] <= sl:
    break
```
Because the data is on a 4-Hour timeframe, a single candle encapsulates significant price action. If a single 4H candle's High exceeds the TP and its Low exceeds the SL, the code unconditionally assumes the TP was hit first. In high-volatility regimes (like crypto), it is entirely possible the SL was hit in the first 5 minutes of the 4H window. This look-ahead bias artificially inflates the win rate.

## 3. Feature Engineering & Selection
Using `add_all_ta_features` is a naive "kitchen-sink" approach. 
- **Multicollinearity:** It dumps dozens of highly correlated features (e.g., various moving averages, MACDs) into the model. While trees are somewhat robust to multicollinearity, it increases dimensionality and the risk of overfitting.
- **Lack of Stationarity:** Many TA features (like raw indicator values) are not stationary. Tree-based models struggle to extrapolate beyond the feature bounds seen in training, making them fragile to new price regimes.
- **No Feature Importance/Ablation:** There is no analysis of which features actually contribute to the model's predictive power.

## 4. Validation Methodology
- **Single Hold-out Split:** A simple 80/20 chronological split is inadequate for financial time series. Market regimes shift over time. 
- **Missing Walk-Forward Validation:** The pipeline lacks Walk-Forward Validation (WFV) or Purged K-Fold Cross-Validation (a la Marcos Lopez de Prado). Without testing across multiple disjoint periods with purging and embargoing, the results are highly susceptible to regime-specific overfitting.

## 5. Hyperparameter Optimization & Model Selection
- **Hardcoded Parameters:** The XGBoost hyperparameters (`max_depth=3`, `learning_rate=0.05`, `subsample=0.8`) are hardcoded. While the author claims this is "Strict regularization to prevent overfitting," this is an arbitrary assertion. 
- **Lack of Tuning:** There is no hyperparameter search (e.g., GridSearch, Bayesian Optimization via Optuna) on a validation set to establish an optimal, robust configuration.

## 6. Evaluation Metrics
- **Misaligned Objectives:** The model is optimized for `logloss`, but the evaluation is purely based on a hard probability threshold of `0.50` predicting a binary win rate.
- **Ignoring Model Calibration:** If trading size or conviction is tied to probability, the model must be calibrated. The script does not measure Brier score or plot calibration curves.

## 7. The Synthetic Data Fallacy
If real data is missing, the script generates a cumulative sum of normal random variables (a pure random walk). By definition, a random walk has no momentum or mean-reverting memory. Training an XGBoost model on this data to find a directional edge is mathematically futile; any non-zero edge discovered on the test set is pure statistical noise/overfitting.

## Verdict
The ML pipeline is fundamentally broken due to data leakage, intra-candle look-ahead bias, and inadequate validation schemes. The results generated by this pipeline cannot be trusted.
